# 技术选型与方案对比总结

更新时间：2026-03-17

## 一、背景

本文档记录了从零开始搭建个人文学博客的完整过程，包括技术选型讨论、核心概念解释、方案对比以及最终实现。

目标：搭建一个可以公网访问的文学类博客网站，用于发布文学解说、书籍解读等内容。

---

## 二、技术选型讨论

### 2.1 最初的需求

用户最初的需求是：
1. 想做一个个人网站，有一个参考案例
2. 主要内容是文学解说、书籍解读
3. 希望有 RSS 订阅功能
4. 不想使用之前 Halo + 腾讯云的方案（太麻烦、太花钱）

### 2.2 可选方案对比

| 方案 | 优点 | 缺点 | 费用 |
|------|------|------|------|
| **Hexo + GitHub Pages** | 免费、零维护、主题丰富 | 需要改配置文件 | 0 元/年 |
| **VitePress + GitHub Pages** | 现代化、性能好 | 主题较少、偏技术风 | 0 元/年 |
| **Hugo + Vercel** | 速度快、免费部署 | Vercel 有一定门槛 | 0 元/年 |
| **Halo + 腾讯云** | 有后台管理、操作方便 | 需要服务器、需要备案 | 数百元/年 |
| **Ghost（托管版）** | 原生支持 RSS、体验好 | 需要付费 | $9/月起 |

### 2.3 最终选择

**Hexo + GitHub Pages + GitHub Actions**

理由：
1. Hexo 学习曲线平缓，中文社区活跃
2. GitHub Pages 完全免费，无需服务器
3. GitHub Actions 自动化部署，推送即更新
4. 支持 RSS 订阅
5. 总费用：0 元

### 3.2 各组件职责

| 组件 | 职责 | 比喻 |
|------|------|------|
| **Git** | 版本控制、代码推送 | 邮递员 |
| **GitHub** | 代码托管平台 | 仓库 |
| **Hexo** | 静态网站生成器（Markdown → HTML）| 排版印刷厂 |
| **GitHub Actions** | 自动化工具（按剧本执行任务）| 流水线工人 |
| **GitHub Pages** | 网站托管服务 | 书店/图书馆 |
| **gh-pages 分支** | 存放生成好的网页 | 书架 |

### 3.3 什么是 GitHub Pages？

GitHub Pages 是 GitHub 免费提供的网站托管服务。

- **普通仓库**：存代码，别人只能下载
- **GitHub Pages**：把代码变成网站，别人可以访问 URL

关键配置：
1. Source 选择 `Deploy from a branch`
2. Branch 选择 `gh-pages` 分支
3. Directory 选择 `/ (root)`

这样 GitHub Pages 就会把 gh-pages 分支根目录的文件当成网站展示出去。

### 3.4 什么是 GitHub Actions？

GitHub Actions 是 GitHub 免费提供的自动化工具。

工作原理：
1. 在 `.github/workflows/` 目录下放置 yml 配置文件
2. 当触发条件满足时（如 push 到 main 分支）
3. GitHub 自动找一台服务器执行 yml 里定义的步骤

本项目的 `deploy.yml` 做了这些事：
```yaml
# 每次推送到 main 分支时触发
on:
  push:
    branches: [main]

jobs:
  deploy:
    steps:
      - 拉取代码
      - 安装 Node.js
      - npm install（安装依赖）
      - hexo generate（生成网页）
      - 推送到 gh-pages 分支
```

### 3.5 什么是分支（Branch）？

把分支想象成**平行宇宙**：

```
main 分支（主宇宙）           gh-pages 分支（平行宇宙）
├── source/_posts/           ├── index.html
├── _config.yml              ├── css/
├── package.json             ├── js/
└── node_modules/            └── 2024/03/文章/
     ↓                            ↓
  源代码                       生成的网页
```

- **main 分支**：存放源代码（Markdown、配置文件）
- **gh-pages 分支**：存放生成的网页（HTML、CSS）

两个分支互不干扰，但存在于同一个仓库里。

### 3.6 node_modules 是什么？

`node_modules/` 是 npm 安装依赖的目录，里面是**别人写好的代码**。

```
node_modules/
├── hexo/                    ← Hexo 核心代码
├── hexo-theme-butterfly/    ← Butterfly 主题代码
├── hexo-renderer-marked/    ← Markdown 解析器
└── 数百个其他包...
```

**关键点**：
- `node_modules/` 不需要推送到 GitHub（在 `.gitignore` 里排除）
- GitHub Actions 会在构建时通过 `npm install` 重新安装
- 你只需要关注 `source/_posts/` 里的文章

## 四、动态博客 vs 静态博客

### 4.1 对比

| 对比项 | 动态博客（Halo/WordPress） | 静态博客（Hexo/Hugo） |
|--------|---------------------------|----------------------|
| **后台管理** | 有，网页登录管理 | 没有，本地写文件 |
| **数据库** | 需要 | 不需要 |
| **服务器** | 需要运行程序 | 不需要 |
| **写文章** | 后台编辑器 | Markdown 文件 |
| **改菜单** | 后台点一点 | 改配置文件 |
| **装插件** | 后台点击安装 | npm install |
| **费用** | 需要服务器费用 | 可以完全免费 |

### 4.2 为什么动态博客需要服务器？

| 功能 | 动态博客实现方式 | 静态博客实现方式 |
|------|-----------------|-----------------|
| **存文章** | 存数据库 | 存 Markdown 文件 |
| **显示文章** | 实时从数据库读取 | 提前生成 HTML |
| **改配置** | 后台修改，实时生效 | 改配置文件，重新生成 |

**动态博客**：每次有人访问，都要运行程序去查数据库 → 需要服务器
**静态博客**：提前把所有页面生成好 → 不需要服务器

### 4.3 选择建议

| 场景 | 推荐方案 |
|------|----------|
| 不想碰代码，想要后台管理 | Halo + 服务器 |
| 能接受改配置文件，想要免费 | Hexo + GitHub Pages |
| 追求极致性能 | Hugo + GitHub Pages |

## 五、新旧方案对比

### 5.1 旧方案：Halo + 腾讯云

**配置要求**：
- 服务器：至少 2核4G
- 域名：购买 + 备案（约 1 个月）
- 维护：需要运维知识

**遇到的问题**：
1. 服务器配置小了带不动，数据库打不开
2. 内存太小被冲爆，升级需要多花钱
3. 域名备案花时间
4. 一年维护费用数百到数千元

### 5.2 新方案：Hexo + GitHub Pages

**配置要求**：
- 服务器：不需要
- 域名：github.io 免费
- 维护：零维护

**优势**：
1. 完全免费
2. 无需备案
3. 自动 HTTPS
4. 全球可访问
5. 推送即更新

### 5.3 费用对比

| 项目 | 旧方案（腾讯云） | 新方案 |
|------|-----------------|--------|
| 服务器 | 数百元/年 | 0 元 |
| 域名 | 几十元/年 | 0 元 |
| 备案 | 时间成本 | 无需备案 |
| CDN | 额外费用 | GitHub 自带 |
| **总计** | **数百元/年** | **0 元/年** |

## 六、常见问题解答

### 6.1 为什么我改了配置但网页没变？

可能原因：
1. 本地 `hexo server` 旧进程仍在运行
2. 浏览器缓存没有刷新
3. GitHub Actions 还没完成部署

解决方法：
1. 停止旧服务，重新 `hexo clean && hexo generate && hexo server`
2. 强制刷新浏览器（Ctrl+F5）
3. 等待 GitHub Actions 完成（约 1-2 分钟）

### 6.2 为什么网站没有样式？

这是 GitHub Pages 子路径站点的常见问题。

如果网站地址是 `https://用户名.github.io/仓库名/`，需要在 `_config.yml` 配置：

```yaml
url: https://用户名.github.io/仓库名
root: /仓库名/
```

否则资源路径会是 `/css/xxx.css` 而不是 `/仓库名/css/xxx.css`。

### 6.3 以后怎么更新文章？

```bash
# 1. 在 source/_posts/ 下新建 .md 文件写文章

# 2. 推送到 GitHub
cd 项目目录
git add .
git commit -m "新增文章：xxx"
git push

# 3. 等待 1-2 分钟，自动部署完成
```

### 6.4 需要买域名吗？

**不需要**。`github.io` 域名：
- 永久免费
- 自动 HTTPS
- 无需备案
- 全球可访问

只有想要个性化域名（如 `yourname.com`）时才需要花钱购买。

## 九、总结

### 9.1 方案优势

1. **零成本**：不需要服务器、不需要域名
2. **零维护**：没有服务器运维负担
3. **零门槛**：在 AI 帮助下可以完成
4. **全自动**：推送代码，自动部署

### 9.2 适合场景

- 个人博客
- 作品集
- 文档站
- 读书笔记
- 任何静态内容网站

### 9.3 不适合场景

- 需要用户登录
- 需要数据库
- 需要后端计算
- 需要实时交互
