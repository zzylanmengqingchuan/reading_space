# 阑梦读书会项目技术总结报告

更新时间：2026-03-17

## 一、项目背景与目标

这个项目是一个基于 Hexo 搭建的个人博客站点，主题使用 `hexo-theme-butterfly`。项目最初由其他模型生成，定位为文学、历史、读书笔记类型的中文博客。后续在实际使用过程中，围绕以下几个方向进行了连续调整：

1. 站点定位和品牌信息重设
2. 首页结构与视觉风格重做
3. 作者信息与 About 页面重构
4. 文章页信息精简
5. 本地字体接入
6. GitHub Actions 自动部署链路修复
7. GitHub Pages 子路径站点资源加载修复
8. 首页列表展示逻辑与时间显示逻辑调整

这份文档的目标是把这次对话中涉及到的关键问题、排查思路、配置修改点、技术原理和后续维护建议，完整记录下来，方便以后：

1. 用户自己回查
2. 交给其他模型继续接手
3. 后续扩展内容发布和自动部署时减少重复排查

## 二、项目原始结构与技术栈

项目目录是一个标准 Hexo 站点，核心文件如下：

1. `package.json`
   定义了脚本：
   - `npm run build` -> `hexo generate`
   - `npm run clean` -> `hexo clean`
   - `npm run server` -> `hexo server`

2. `_config.yml`
   这是 Hexo 主配置文件，负责：
   - 站点标题、描述、作者
   - 网址和根路径
   - permalink 规则
   - 时间格式
   - 主题选择
   - feed 配置
   - deploy 配置

3. `_config.butterfly.yml`
   这是 Butterfly 主题配置文件，负责：
   - 导航菜单
   - 首页文章展示方式
   - 侧边栏显示内容
   - 文章元信息
   - 文章版权、分享、分页等模块
   - 右侧按钮与主题行为

4. `source/_posts`
   存放 Markdown 文章。Hexo 会把这里的 Markdown 编译成 `public` 里的 HTML。

5. `source/about/index.md`
   About 页面内容。

6. `source/css/custom.css`
   自定义样式文件，用于覆盖 Butterfly 的默认样式。

7. `.github/workflows/deploy.yml`
   GitHub Actions 自动部署工作流文件。

主题不是本地源码修改版，而是通过 `node_modules/hexo-theme-butterfly` 提供，因此后续大多数改动都应优先走：

1. 站点配置覆盖
2. 自定义 CSS 覆盖
3. 内容页结构重写

而不是直接改 `node_modules` 内的主题源码。

## 三、最初发现的问题

在第一次扫描项目时，确认了项目可以正常 `hexo generate`，说明站点本身能运行。但同时也发现了几个典型问题：

   站点标题“墨香书斋”占据整屏，导致文章必须下滚才能看到。

2. 导航过重
   菜单里有中国文学、外国文学、诗歌鉴赏、读书笔记、归档、关于等多个入口，但用户实际并不需要。

3. 站点身份信息是占位内容
   包括：
   - `your-username`
   - `your-email@example.com`
   - 默认 GitHub 链接
   - 默认 About 文案

4. 首页展示方式不符合用户使用习惯
   用户希望首页更像索引页，而不是“文章摘要卡片墙”。

5. 默认示例文章混在内容里
   例如：
   - `Hello World`
   - 《百年孤独》……
   - 《红楼梦》……
   - 《定风波》……

这些不属于用户自己的真实内容。

## 九、首页列表展示逻辑的多轮修正

这是整个过程中争议最大的一块，也是用户最不满意的一块。

### 9.1 第一阶段：卡片式文章流

最开始首页沿用 Butterfly 默认的“摘要卡片”模式：

1. 标题
2. 元信息
3. 摘要
4. 大块卡片容器

用户明确不接受，因为：

1. 每篇文章占空间太大
2. 一屏只能放很少内容
3. 本质上不像博客索引，而像宣传卡片

### 9.2 第二阶段：压缩卡片但仍保留卡片骨架

曾经尝试通过 CSS 把卡片改瘦，隐藏摘要，只保留标题和时间。
但由于 Butterfly 首页默认样式里：

1. `recent-post-item` 自带固定高度
2. `recent-post-info` 自带大块 padding
3. 整个首页骨架仍按卡片布局走

所以实际结果变成：

1. 框没有了
2. 但是大面积空白仍然存在

这就是用户指出“你只是把框去掉了，空间还是那么大”的根本原因。

### 9.3 第三阶段：真正转向目录式列表

后续的修正目标明确变成：

1. 每篇文章只占一行
2. 一个项目就是一个标题入口
3. 时间不要另起一行
4. 不要“发表于”“更新于”
5. 不要显示摘要

因此首页展示开始被强制压成：

1. 标题
2. 单一日期
3. 双列索引布局

不过这一块仍然值得后续继续打磨。因为 Butterfly 首页默认模板不是为“极简列表页”设计的，如果后面还需要更极致地接近参考博客，最稳妥的方法可能是：

1. 直接覆盖首页模板
2. 或者新增一个专门的首页布局

而不是继续在默认 `recent-post-item` 结构上做层层覆盖。

## 十、文章时间显示问题与修复

用户反馈：刚发的文章却显示“20 小时前”，观感非常差。

### 10.1 原因

并不是文章时间写错了，而是主题配置用了“相对时间”：

1. `date_format: relative`
2. `date_type: both`

所以页面会显示：

1. 多少分钟前
2. 多少小时前
3. 更新于……

### 10.2 修复方案

用户要求：

1. 不显示具体几点
2. 不显示“更新于”
3. 只显示一个固定日期

因此做了两处调整：

1. `_config.yml`
   - `date_format` 改为 `YYYY年M月D日`

2. `_config.butterfly.yml`
   - `date_type: created`
   - `date_format: date`

这样最终效果变成：

1. 只显示创建日期
2. 不显示相对时间
3. 不显示更新时间

## 十一、字体切换为霞鹜文楷

用户给出了本地字体目录：

`C:\Users\Administrator\AppData\Roaming\Typora\themes\happysimple\source\font\霞鹜文楷`

目标是把站点字体切换为“霞鹜文楷”。

### 11.1 处理方式

由于直接引用 Typora 的本地路径不适合网站部署，因此采取了正确的工程化方案：

1. 读取字体目录
2. 把字体文件复制进项目
3. 在项目内定义 `@font-face`
4. 站点统一使用项目内字体

最终复制了：

1. `source/font/LXGWWenKai-Regular.ttf`
2. `source/font/LXGWWenKai-Bold.ttf`

并在 `source/css/custom.css` 中定义：

1. `@font-face`
2. `font-family: "LXGW WenKai"`

### 11.2 为什么不能直接用 Typora 本地路径

因为网站部署到 GitHub Pages 后，浏览器访问的是线上资源，而不是你的电脑本地磁盘。只有把字体文件放进项目并随站点一起发布，线上页面才能正常加载。

## 十三、文章新增机制说明

用户关心后续怎么继续加文章。

### 13.1 正确目录

所有新文章都应放在：

`source/_posts`

### 13.2 文件格式

每篇文章需要是 Markdown 文件，并带有 front-matter，例如：

```md
---
title: 文章标题
date: 2026-03-17 10:00:00
tags:
  - 标签1
  - 标签2
---
```

### 13.3 排序逻辑

Hexo 首页默认按 `date` 倒序：

1. 新文章排前面
2. 旧文章自动往后

这不是人工写死顺序，而是框架自身逻辑。

### 13.4 腾讯云图床是否可用

可以，只要图片 URL 是公网可访问链接，例如：

```md
![说明](https://你的图床域名/xxx.jpg)
```

需要满足：

1. 链接可以直接访问
2. 没有防盗链限制当前域名
3. HTTPS 可正常访问

## 十四、GitHub Actions 自动部署链路与报错修复

用户项目里配置了 GitHub Actions，希望做到：

1. 在 `source/_posts` 下新增 Markdown
2. `git push`
3. GitHub Actions 自动构建
4. 自动部署到 GitHub Pages

### 14.1 工作流逻辑

工作流文件：

`.github/workflows/deploy.yml`

它的职责是：

1. 拉取仓库代码
2. 安装 Node.js
3. 安装依赖
4. 执行 `hexo clean`
5. 执行 `hexo generate`
6. 把 `public` 推送到 `gh-pages`

### 14.2 当时的报错

用户截图显示：

`Unable to resolve action peaceiris/actions-ghpages, repository not found`

### 14.3 根本原因

Action 名称写错了：

错误：

`peaceiris/actions-ghpages@v4`

正确：

`peaceiris/actions-gh-pages@v4`

少了一个连字符，导致 GitHub 在工作流初始化阶段就失败。

### 14.4 修复内容

工作流中还顺手补了：

1. `permissions: contents: write`
   允许 Actions 把产物推送到 `gh-pages`

2. `cache: 'npm'`
   利用 `actions/setup-node` 的缓存能力

3. `npm ci`
   比 `npm install` 更适合 CI 环境

## 十五、GitHub Pages 样式丢失的真正原因

这是后续最关键的一次线上问题。

用户访问：

`https://zzylanmengqingchuan.github.io/reading_space/`

结果页面只剩裸 HTML，没有任何样式。

### 15.1 现象

页面内容在，说明 HTML 成功部署了。  
但 CSS、JS、图片都没加载。

### 15.2 根本原因

这是 GitHub Pages 子路径站点的典型问题。

项目部署地址是：

`/reading_space/`

但 Hexo 配置里原来还是默认：

1. `url: https://your-username.github.io`
2. 没有正确设置 `root`

结果生成出来的 HTML 资源地址是：

1. `/css/index.css`
2. `/js/main.js`
3. `/img/avatar.png`

这些路径会去域名根目录找资源，而不是去 `/reading_space/` 下找。

### 15.3 修复方法

修改 `_config.yml`：

1. `url: https://zzylanmengqingchuan.github.io/reading_space`
2. `root: /reading_space/`

并同步修正：

1. `_config.butterfly.yml` 中自定义 CSS 的注入路径
2. `custom.css` 中字体文件路径，改成相对路径，避免子目录下引用错误

修复后生成结果里已经变成：

1. `/reading_space/css/index.css`
2. `/reading_space/css/custom.css`
3. `/reading_space/js/main.js`
4. `/reading_space/img/avatar.png`

这就是线上样式恢复的关键。

## 十六、为什么本地构建时会遇到若干权限与进程问题

在整个过程中，还遇到了几类容易反复出现的环境问题。

### 16.1 4000 端口被占用

表现：

`EADDRINUSE: address already in use :::4000`

原因：

之前的 `hexo server` 进程没停掉。

排查方式：

```powershell
netstat -ano | findstr :4000
```

结束旧进程：

```powershell
taskkill /PID 进程号 /F
```

然后再：

```powershell
npm run server
```

### 16.2 某些进程无法从当前权限结束

有几次尝试 `Stop-Process` 时被系统拒绝访问。  
这通常说明：

1. 进程权限更高
2. 当前会话没权限直接终止

这种情况下更稳的做法还是手动用 `taskkill /PID ... /F`

### 16.3 npm cache 权限异常

出现过：

1. `EPERM`
2. `EACCES`
3. 本地 `npm ci` 无法访问缓存目录

这类错误是本机环境权限问题，不是 Hexo 项目逻辑问题。  
后来通过重新安装依赖恢复了 `node_modules`，继续完成生成。

9. GitHub Pages 子路径资源问题已修复
10. 时间显示改为固定日期，不再显示相对时间

## 十八、后续最值得继续优化的地方

虽然已经解决了绝大多数结构性问题，但还有一块值得继续优化：

### 首页列表模板

用户最终希望首页非常接近如下形式：

`2026-03-15 文章标题`

或者：

`2026年3月17日 文章标题`

当前首页虽然已经被持续压缩，但由于 Butterfly 默认首页模板天然是卡片流结构，所以若要做到完全极简，下一步建议：

1. 直接改首页模板
2. 或新增一个自定义首页 layout
3. 不再继续“覆盖默认卡片结构”

这样可以一次性彻底解决：

1. 空白过大
2. 排列不够像索引页
3. 样式覆盖过多

## 十九、给后续模型的接手建议

如果后续让其他模型继续接手，最重要的是先理解这几个前提：

1. 这是一个 Hexo + Butterfly 项目
2. 站点部署在 GitHub Pages 子路径 `/reading_space/`
3. 不要随便把 `url` 和 `root` 改回去
4. 不要恢复默认侧边栏和文章页底部模块
5. 首页目标是“目录索引页”，不是文章摘要卡片流
6. About 页面才是作者信息承载页
7. 字体现在依赖项目里的本地 `LXGWWenKai` 文件，不要再改回外链字体

如果要继续优化，优先级建议如下：

1. 优先处理首页列表模板重构
2. 再处理视觉细节
3. 最后再考虑评论、搜索、归档等增强功能
